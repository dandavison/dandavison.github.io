Standalone Activity — behavior projections

Every artifact here is generated deterministically from the executable behavior spec (saaspec.Model) — the same spec the conformance tests check the real server against. Nothing on this page is hand-authored behavior, and none of it can drift from the spec.

Lifecycle & operations diagram

What each operator action does in each phase of an activity's life: immediate, deferred to the next attempt boundary, or not permitted.

Interactive explorer →

Lay an activity down along a time axis, choose its start delay (which shifts the timeouts), and send operations — each styled up-front as allowed / deferred / not-permitted. An attempt-boundary grid shows how a pending pause / reset / cancel resolves at the end of an attempt, versus no pending request.

Animated timelines →

Each trace sweeps along a timeline through its timer-window phases, with a reusable animateTrace primitive. Comparative traces show what happens at the end of an attempt with and without a pending pause / reset / cancel (e.g. a pending cancel suppresses the retry).
